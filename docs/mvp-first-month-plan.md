## ⚠️ 风险预案

### 风险 1：注册增长不达标

**触发条件**：Day 7 新增注册 < 100，或 Day 14 < 300

**根本原因分析**：
- guide.md 难度过高？
- 邀请机制不清楚？
- 渠道选择不对？
- 游戏缺乏趣味性？

**应对方案**（按优先级）：

1. **紧急 guide.md 迭代**（Day 2-3 内）
   - 收集 5-10 个新用户的反馈，精简步骤
   - 加入更多例子和变体命令
   - 测试不同 AI 工具的兼容性

2. **扩大渠道投放**（Day 4）
   - 联系 5-10 个有影响力的博主，申请转发
   - 发布在论坛、社区（如 Reddit、产品猎人、HackerNews）
   - 邀请 Discord 等社区的版主做转发

3. **邀请激励加码**（Day 6）
   - 讨论是否提前开放「邀请 TOP 10 得额外奖励」
   - 如 MVP 能承受，可小幅激励（如 10 邀请 +2 EGGS）

4. **优化新手体验**（Week 2）
   - 如注册数仍未起色，考虑做注册新手教程
   - 加入游戏内提示（如：「你可以邀请朋友来一起养鸡」）

### 风险 2：DAU 和留存率低

**触发条件**：DAU/注册 < 15%，或 7 日留存 < 15%

**根本原因分析**：
- 早报不够吸引人？
- 产蛋速度太慢？
- 邀请难度太高？
- 没有紧急感（不知道母鸡什么时候死）？

**应对方案**（按优先级）：

1. **早报内容优化**（Day 4-5）
   - 进行 A/B 测试，对比 2-3 个早报版本
   - 加入更多数据可视化（emoji、图表）
   - 突出邀请排行，激发竞争感

2. **产蛋机制调整**（Week 2，需谨慎）
   - 如果确实太慢，考虑改为 12h 产 1 个（但要平衡游戏经济）
   - 或增加「早报奖励」：每日调用早报 +0.5 EGGS

3. **死亡预警优化**（Week 3）
   - Day 27-29 提前预警，增加紧急感
   - 文案强调：「母鸡快要停产，快邀请新鸡吧」

4. **邀请机制重设计**（Week 3-4）
   - 降低邀请门槛（但不在 MVP 阶段做）
   - 或直接赠送邀请机会给新手

### 风险 3：被恶意刷号

**触发条件**：
- 同一 IP 短时间内出现多个账户
- 邀请排行中同一用户邀请数异常高
- 发现批量垃圾用户名

**实时防护**（已在 MVP 中内置）：
- 同 IP 限流（1h 最多 5 个账户）
- 邀请限流（24h 最多 10 邀请）
- 用户名、邮箱唯一性校验

**应急处理**：

1. **检测与隔离**（即时）
   - 数据库查询可疑账户（邀请数异常、创建时间相同等）
   - 标记为「待审核」账户，停止其邀请功能

2. **人工审核**（Day 内）
   - 人工验证这些账户是否真实
   - 可联系邀请者进行确认

3. **数据修复**（Day 内）
   - 删除明确的垃圾账户
   - 回滚虚假邀请关系
   - 重新计算排行榜

4. **防护加强**（Day 1-2）
   - 加强 IP 限流阈值（如改为 1h 2 个）
   - 引入 CAPTCHA 防机器人
   - Week 3 开发更复杂的反作弊逻辑

5. **沟通与透明**
   - 公告说明处理情况
   - 保护真实用户的权益

### 风险 4：服务器压力超预期

**触发条件**：
- API 响应时间 > 2s
- 数据库连接数 > 80%
- 频繁出现超时错误

**实时监控**（已配置）：
- 告警阈值：错误率 > 5%、响应时间 > 2s

**应急扩展**：

1. **即时扩容**（1h 内）
   - 增加数据库连接池
   - 增加应用服务器实例
   - 启用 CDN 缓存静态资源

2. **流量控制**（2h 内）
   - 降级非关键功能（如排行榜可延迟 1h 更新）
   - 实施用户请求限流
   - 关闭不必要的功能

3. **性能优化**（Day 内）
   - 数据库查询优化
   - 缓存策略调整
   - 异步任务队列（如排行榜计算移到后台）

4. **预防方案**（Week 2-3）
   - 提前扩容（预留 50% 容量）
   - 做压力测试（模拟 1000+ 并发）
   - 考虑多地域部署
